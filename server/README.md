# Rooms API

Первый работающий серверный модуль Rooms. Он отделён от статического сайта, чтобы GitHub Pages продолжал работать во время поэтапного переноса сценариев на backend.

## Что реализовано

- `GET /health` — состояние процесса и активного хранилища;
- `GET /v1/cities` — поддерживаемые города;
- `GET /v1/cities/{cityId}/stats` — количество площадок и помещений, а также безопасный диапазон активной аудитории;
- `GET /v1/rooms` — каталог с фильтрами по городу, гостям, типу, возможностям и цене;
- `GET /v1/rooms/{roomId}` — карточка по UUID или старому slug;
- `GET /v1/rooms/{roomId}/reviews` — одобренные отзывы без контактов и внутренних идентификаторов клиента;
- `POST /v1/availability/search` — общие свободные окна одного или нескольких помещений;
- `POST /v1/auth/client/register` и `POST /v1/auth/login` — регистрация и вход клиента по email или телефону;
- `POST /v1/auth/refresh`, `POST /v1/auth/logout` и `GET /v1/me` — ротация сессии, безопасный выход и текущий профиль;
- `POST /v1/auth/password-reset/request` и `/confirm` — одноразовая 15-минутная ссылка, смена пароля и отзыв всех прежних сессий;
- `GET /v1/me/sessions`, `DELETE /v1/me/sessions/{sessionId}` и `/others` — список активных устройств и управляемый выход;
- `PATCH /v1/me` — изменение имени, контактов и города; смена пароля требует текущий пароль;
- `GET/PATCH /v1/me/notification-settings` — серверные настройки email и Telegram для любого авторизованного кабинета;
- `POST /v1/me/notification-settings/test` и `GET /v1/me/notification-deliveries` — тест канала и личный журнал с замаскированными адресами;
- `POST /v1/bookings` — проверка состава, вместимости и свободного окна, серверный расчёт помещений, услуг, предоплаты и создание заявки одной транзакцией;
- `GET /v1/bookings` — заявки авторизованного клиента с фильтром по группе статусов;
- `GET/PATCH /v1/partner/venue` — карточка, контакты и недельный график назначенной площадки; публичные изменения уходят на модерацию;
- `GET /v1/partner/moderation` — история проверок своей площадки, включая решение и комментарий Rooms;
- `GET/POST /v1/partner/rooms` и `PATCH /v1/partner/rooms/{roomId}` — помещения, услуги, рабочие часы и буфер только своей площадки;
- `PUT/DELETE /v1/partner/schedule-exceptions/{date}` — особые часы, закрытие дня и возврат к недельному графику;
- `GET /v1/partner/bookings` — защищённая очередь заявок назначенной площадки;
- `GET /v1/partner/reservations` — ручные брони и техническая занятость только своей площадки с фильтрами по помещению и датам;
- `POST /v1/partner/reservations` и `PATCH /v1/partner/reservations/{reservationId}` — создание и перенос записи с повторной проверкой графика, буфера и конфликтов;
- `POST /v1/partner/reservations/{reservationId}/cancel` и `/restore` — отмена и безопасное восстановление ручной записи;
- `DELETE /v1/partner/reservations/{reservationId}` — удаление только технического блока; ручная бронь сохраняется в истории отменённой;
- `POST /v1/partner/bookings/{bookingId}/confirm` — подтверждение партнёром и удержание всех выбранных помещений на 15 минут;
- `POST /v1/partner/bookings/{bookingId}/reject` — отклонение с причиной и освобождение неоплаченного удержания;
- `POST /v1/partner/bookings/{bookingId}/proposal` — проверка общего свободного окна и серверный пересчёт альтернативного времени;
- `POST /v1/bookings/{bookingId}/proposal/accept` и `/decline` — ответ клиента; принятие атомарно удерживает все помещения на 15 минут;
- `GET/POST /v1/bookings/{bookingId}/messages` — переписка клиента и площадки с отметками прочтения и блокировкой контактов до предоплаты;
- `POST /v1/bookings/{bookingId}/payment-intent` — серверный платёж на неизменяемую клиентом сумму предоплаты;
- `POST /v1/payments/{paymentId}/demo-complete` — локальное идемпотентное завершение оплаты и превращение удержания в постоянный резерв;
- `GET /v1/admin/moderation` — защищённая очередь публичных правок площадок и помещений;
- `POST /v1/admin/moderation/{moderationId}/approve` и `/reject` — атомарное применение или отклонение правок с аудитом и комментарием партнёру;
- `GET /v1/admin/notification-deliveries` — защищённый журнал серверной очереди с фильтрами по каналу и статусу;
- worker уведомлений ставит в очередь события регистрации, восстановления пароля, заявок, чата, предложений времени, подтверждений, отказов и предоплаты;
- тела email/Telegram зашифрованы AES-256-GCM, уникальные ключи не допускают повторной доставки одного события, после отправки содержимое очищается;
- пароли в PostgreSQL хешируются через `argon2id`, refresh-токены хранятся только в виде SHA-256 и передаются в `HttpOnly` cookie;
- единый формат ошибок с `requestId`;
- CORS только для локального запуска и GitHub Pages Rooms;
- два взаимозаменяемых хранилища каталога: память для демо и PostgreSQL для рабочего режима;
- чтение из PostgreSQL фотографий, услуг, расписания, отзывов и занятых интервалов;
- автоматические API-тесты через `fastify.inject`.

## Запуск

Требуется Node.js 24 LTS или новее.

```powershell
npm install
npm run dev
```

Сайт и API откроются на `http://127.0.0.1:3001`. При таком запуске frontend автоматически получает каталог и отзывы из PostgreSQL. Если API недоступен, публичный GitHub Pages продолжает использовать встроенный каталог.

```powershell
npm test
npm run typecheck
npm run build
npm run smoke:partner-catalog
npm run smoke:auth-security
npm run notifications:run-once
npm start
```

Для локальных настроек можно создать `.env` по образцу `.env.example`. Файл `.env` исключён из Git.

Для режима PostgreSQL обязательно задайте `AUTH_TOKEN_SECRET` длиной не менее 32 байт. В локальном HTTP-preview оставьте `AUTH_COOKIE_SECURE=false`; на рабочем HTTPS-домене переключите значение на `true`.

`ENABLE_DEMO_PAYMENTS=true` включает локальный платёжный адаптер без передачи карточных реквизитов в Rooms. В production он по умолчанию выключен и должен быть заменён серверным адаптером банка и подписанным webhook.

`EXPOSE_PASSWORD_RESET_TOKEN=true` показывает одноразовую ссылку прямо в локальном интерфейсе, пока почтовый провайдер не подключён. В production этот режим по умолчанию выключен: токен нельзя возвращать клиенту, его должен доставлять отдельный email/SMS worker.

`NOTIFICATION_DELIVERY_MODE=log` обрабатывает очередь без внешней отправки и не пишет содержимое сообщений в журнал. Для реальной доставки задайте `live`, отдельный `NOTIFICATION_ENCRYPTION_KEY` длиной не менее 32 байт, `SMTP_URL` с `EMAIL_FROM` и/или `TELEGRAM_BOT_TOKEN`. `NOTIFICATION_WORKER_ENABLED=false` отключает фоновый цикл; одиночную пачку можно обработать командой `npm run notifications:run-once`. Пользователь Telegram должен сначала открыть чат с ботом, после чего в настройках указывается числовой chat ID; запись `@channel` подходит для канала, где бот имеет право публикации.

Без `DATABASE_URL` сервер использует `MemoryCatalogRepository`. Если `DATABASE_URL` заполнен, соединение с PostgreSQL обязательно: при ошибке сервер завершит запуск, а не подменит рабочие данные демо-каталогом.

## PostgreSQL

После установки PostgreSQL создайте пользователя и базу `rooms`, затем укажите адрес в `.env`:

```text
DATABASE_URL=postgresql://rooms:ваш-пароль@localhost:5432/rooms
DATABASE_SSL=disable
```

Применить все ещё не выполненные миграции по порядку:

```powershell
npm run db:migrate
```

Команда хранит имя и SHA-256 применённой миграции в `schema_migrations`. Повторный запуск пропустит ту же версию, а изменённую миграцию остановит с ошибкой.

Для локальной разработки можно явно загрузить текущий демо-каталог:

```powershell
$env:ALLOW_DEMO_SEED="true"
npm run db:seed
```

Заполнение обновляет только фиксированные демо-площадки Rooms, фотографии, услуги, пять связанных демо-отзывов, тестовый кабинет партнёра Kids Loft и администратора `admin@rooms.ru`. Пароль администратора задаётся через `DEMO_ADMIN_PASSWORD`, в локальном примере используется `rooms2026`. Каждый отзыв создаётся вместе с клиентом и завершённой бронью, чтобы сохранялось правило «отзыв только после посещения». В рабочей базе запускать эту команду нельзя.

## Граница текущего этапа

`MemoryCatalogRepository` возвращает типизированные демо-данные, а `PostgresCatalogRepository` реализует тот же контракт для нормализованной схемы. На текущем компьютере PostgreSQL 17.10 установлен, миграции `0001_initial`–`0005_notification_worker` применяются последовательно, а локальный API запускается с хранилищем `postgresql`.

В публичной статистике точное число клиентов не возвращается. До 20 активных клиентов город имеет статус `launching`, затем API показывает только ступени `20+`, `50+`, `100+`, `200+`, `500+` и `1 000+`. Точные показатели предназначены для защищённой админки.

Сервер уже обрабатывает регистрацию клиента, вход, восстановление пароля, список и отзыв активных сессий, изменение профиля, создание заявки, списки заявок клиента и партнёра, подтверждение и отклонение партнёром, альтернативное время, переписку и внешние уведомления. Подтверждение или принятие предложения создаёт 15-минутное удержание в `room_reservations`; ограничение PostgreSQL и последовательная блокировка помещений не позволяют занять пересекающийся слот. Истёкшее удержание переводит заявку в `expired` и освобождает время. Сообщения хранят отметки прочтения для обеих сторон, а сервер не принимает прямые контакты до статуса `paid`. Ручные брони и технические блоки партнёра записываются в ту же таблицу, учитывают буфер, участвуют в публичной доступности и оставляют историю в `audit_log`. Карточка площадки, комнаты, услуги и расписание восстанавливаются из PostgreSQL; публичные правки лежат отдельно в `moderation_requests`, а решение администратора атомарно применяет данные либо отклоняет их с причиной и записью в аудит. Локальный платёжный адаптер записывает транзакцию в `payment_transactions`, переводит заявку в `paid`, снимает срок резервов и открывает контакты клиента партнёру. Frontend использует эти маршруты в локальном API-preview, а при недоступном API сохраняет статический режим для GitHub Pages. Для production остаются credentials внешних провайдеров, Сбер, подписанный webhook и онлайн-касса.
