# Rooms: переход от frontend-прототипа к backend

Статус: технический план для MVP. Документ не означает, что production backend уже подключён.

## Текущее состояние

- установлен Node.js 24 LTS;
- создан `server` на Fastify и TypeScript;
- работают `/health`, `/v1/cities`, `/v1/cities/{cityId}/stats`, `/v1/rooms`, `/v1/rooms/{roomId}`, `/v1/rooms/{roomId}/reviews` и `/v1/availability/search`;
- публичный каталог, фильтры и пересечение доступности покрыты автоматическими тестами;
- готовы `MemoryCatalogRepository` и `PostgresCatalogRepository` с единым HTTP-контрактом;
- добавлены контролируемая первоначальная миграция и защищённое демо-наполнение;
- PostgreSQL 17.10 установлен как автоматическая служба Windows;
- схема `database.sql` применена миграцией `0001_initial`, демо-каталог загружен;
- локальный API использует `postgresql`; `memory` остаётся только режимом разработки без `DATABASE_URL`;
- frontend-каталог и опубликованные отзывы подключаются к API в локальном preview с автоматическим fallback;
- регистрация клиента, вход по email/телефону, `argon2id`, короткий access token и ротация refresh cookie работают через PostgreSQL;
- профиль клиента и заявки синхронизируются с PostgreSQL; цена, услуги, предоплата и остаток рассчитываются сервером;
- партнёр входит с серверной ролью, видит заявки только своей площадки, подтверждает или отклоняет их через API;
- подтверждение атомарно создаёт 15-минутное удержание всех помещений, а истечение срока освобождает слот;
- GitHub Pages продолжает работать независимо от локального API.

## Цель

Перенести данные из `localStorage` в PostgreSQL, не меняя согласованные пользовательские сценарии:

- клиент выбирает конкретное помещение и свободный интервал;
- новая заявка не блокирует слот;
- подтверждённая заявка блокирует слот на 15 минут ожидания предоплаты;
- оплаченная бронь, ручная бронь, техническая занятость и буфер блокируют доступность;
- телефон клиента открывается партнёру только после предоплаты;
- критичные изменения карточки проходят модерацию;
- все финансовые и административные действия записываются в аудит.

## Рекомендуемый стек

- API: Node.js LTS + Fastify + TypeScript.
- База: PostgreSQL 16 или новее.
- Миграции и запросы: параметризованные запросы `pg`, репозитории и журнал контрольных сумм миграций. ORM можно добавить позже, если объём записывающих сценариев оправдает дополнительный слой.
- Очереди: Redis + BullMQ для email, Telegram, чеков, напоминаний и повторных попыток.
- Файлы: S3-совместимое хранилище с отдельными оригиналами и производными изображениями.
- Авторизация: короткоживущий access token и refresh token в `HttpOnly Secure` cookie.
- Платежи: серверный адаптер Сбер эквайринга и webhook с проверкой подписи.
- Чеки: серверная интеграция с онлайн-кассой и хранение фискальных реквизитов.
- Наблюдаемость: структурированные логи, error tracking и метрики очередей.

## Этапы

### 1. Каркас и чтение каталога

1. [Готово] Реализовать health endpoint и публичное чтение городов и помещений.
2. [Готово] Загрузить типизированные демо-данные Воронежа и Москвы.
3. [Готово] Поднять PostgreSQL и применить `database.sql`.
4. [Готово] Перенести каталог в PostgreSQL через репозиторий без изменения HTTP-маршрутов.
5. [Готово] Добавить серверное чтение одобренных отзывов без контактов клиента.
6. [Готово] Перевести frontend-каталог на API с локальным fallback только для разработки.

### 2. Авторизация и кабинеты

1. [Готово] Создать серверную регистрацию клиента и вход по email/телефону.
2. [Готово] Хранить только хеш пароля (`argon2id`), никогда не переносить пароли из прототипа.
3. [Готово] Реализовать роли `client`, `partner`, `admin`, `accountant` в серверной модели сессии.
4. Добавить восстановление пароля и отзыв всех активных сессий.

### 3. Доступность и бронь

1. [Частично] Чтение графика, исключений, существующих резервов и буферов уже работает; временное удержание записывается сервером, ручная занятость партнёра ещё локальная.
2. [Готово] Перед созданием заявки сервер повторно проверяет общее окно, а сама заявка, помещения, услуги и согласие записываются одной транзакцией.
3. [Готово] При подтверждении создавать временное удержание слота на 15 минут и автоматически освобождать его после истечения срока.
4. При оплате превращать удержание в подтверждённую бронь.
5. [Готово для удержаний] Защититься от двойного подтверждения ограничением БД и последовательной блокировкой строк помещений.

### 4. Оплата и финансы

1. Создавать платёж только на сервере по сумме, рассчитанной из сохранённого заказа.
2. Обрабатывать webhook идемпотентно.
3. Сохранять предоплату, комиссию Rooms, остаток на площадке и сумму партнёру.
4. Формировать возвраты, выплаты и чеки отдельными операциями.

### 5. Уведомления и файлы

1. Перенести очередь из `rooms_notification_outbox_v30` в таблицу `notification_deliveries`.
2. Подключить email provider и Telegram-бота через worker.
3. Загружать фото по подписанным URL, проверять MIME и размер.
4. Создавать горизонтальные и вертикальные производные без потери оригинала.

## Импорт текущего прототипа

1. Войти в админку Rooms.
2. Открыть «Центр миграции данных».
3. Скачать JSON-снимок.
4. Проверить `format=rooms-migration`, `version=1` и контрольные количества коллекций.
5. Импортировать снимок серверной административной командой.
6. Создать приглашения на сброс пароля клиентам и партнёрам.
7. Сверить брони, оплаты, согласия и выплаты до переключения frontend на API.

Снимок намеренно не содержит паролей и активных сессий. Он содержит персональные и финансовые данные, поэтому должен передаваться только по защищённому каналу и удаляться после подтверждённого импорта.

## Обязательные проверки перед запуском

- параллельное подтверждение одного слота не создаёт две брони;
- повторный webhook не создаёт повторную оплату или чек;
- партнёр не видит телефон до предоплаты;
- клиент не видит комиссию Rooms;
- доступ к площадке ограничен её партнёром;
- все административные действия попадают в `audit_log`;
- согласия хранят версию документа, дату, субъект и контекст;
- резервное восстановление PostgreSQL проверено на отдельном окружении.
